# AGENTS.md

## Project

Static website for thelinuxformula.com. No framework, no bundler — HTML5 + Tailwind CSS (CDN) + Vanilla JS. Deployed via GitHub Pages.

## Architecture

- `src/home/index.html` — homepage, sole page entrypoint
- `src/home/main.js` — main thread: instantiates worker, receives messages, updates DOM
- `src/home/newsWorker.js` — Web Worker for async fetch of news data (SHA-256 hash caching, 5min polling)
- `src/raw/` — source Markdown files for news articles (frontmatter: title, date, tags)
- `dist/news/` — build output: `news-feed.json` generated by `scripts/build-news.js`
- `scripts/build-news.js` — MD → JSON build script (Node native modules only, `BUILD_SRC_HOME`/`BUILD_NEWS_SRC`/`BUILD_DIST` env vars for custom paths)
- `prd.md` — product requirements (gitignored, in repo root)
- `prompt/` — sprint prompts (gitignored)
- `scripts/build-news.test.js` — 6 tests for build script (node:test)
- `src/home/newsWorker.test.js` — 6 tests for worker source checks + hash (node:test)
- `src/home/main.test.js` — 6 tests for main thread source checks + HTML container (node:test)

## Commands

```bash
# Build and assemble dist/ (copies HTML/JS + generates news-feed.json)
node scripts/build-news.js

# Run all tests (zero npm dependencies — uses Node 24 built-in test runner)
node --test scripts/build-news.test.js
node --test src/home/newsWorker.test.js
node --test src/home/main.test.js

# Serve locally
python3 -m http.server 8000 --directory dist
```

### Deploy flow

1. Push to `main` triggers GitHub Actions (`.github/workflows/deploy.yml`)
2. CI runs tests → builds `dist/` → uploads as Pages artifact
3. `actions/deploy-pages@v4` deploys the artifact to GitHub Pages
4. GitHub Pages must be configured in Settings → Pages → Source: **GitHub Actions** (not "Deploy from a branch")
5. `dist/` is gitignored on `main` — only exists as a build artifact

## Design system (from `src/home/index.html`)

The reference layout is `src/home/index.html`. All new UI must follow these patterns.

### Color tokens (Material Design 3)

Tailwind config in `index.html` defines 30+ semantic color tokens. Key ones:
- `primary` (#003f87) / `primary-container` (#0056b3) — CTAs, active nav, accent borders
- `surface` (#f8f9fa) — page background
- `surface-container-low/lowest/highest` — card and section backgrounds (layered elevation)
- `on-surface` (#191c1d) — primary text
- `on-surface-variant` (#424752) — secondary/muted text
- `outline` / `outline-variant` — borders and dividers (always at reduced opacity: `/20`, `/50`)
- `error` (#ba1a1a) / `error-container` (#ffdad6) — error states

### Typography (Tailwind `fontFamily` tokens)

- `font-headline` → IBM Plex Serif, serif — headings (`h1`, `h2`, `h3`), brand name
- `font-body` → Inter, sans-serif — body text, descriptions, paragraph content
- `font-label` → IBM Plex Mono, monospace — UI labels, tags, metadata, nav links, small caps

Label text uses `uppercase tracking-widest` (or `tracking-wider`) and `text-xs` / `text-[10px]` sizes.

### Layout patterns

- **Max width:** `max-w-7xl` centered container for nav, main, footer
- **Grid:** 12-column `lg:grid-cols-12` with responsive fallbacks (`grid-cols-1`, `md:grid-cols-*`)
- **Hero:** 8/4 split — `lg:col-span-8` text hero + `lg:col-span-4` diagram/art card
- **Cards:** bento grid sections, `bg-surface-container-low` padded cards with `border-t-4 border-primary` highlights
- **Nav:** sticky top, `backdrop-blur-xl`, transparent overlay (`bg-surface-container-lowest/80`)
- **Footer:** 4-column grid, `font-label` micro-text

### Visual signature rules

- **No rounded corners on interactive elements:** buttons and CTAs use `rounded-none`. `borderRadius` defaults are minimal (0.25rem).
- **Border accent pattern:** `border-l-4` or `border-l-2` for active states and card highlights
- **Grid overlay on body:** 40×40px grid lines via CSS `linear-gradient` background
- **Material Symbols:** always use `material-symbols-outlined` class; filled icons use `style="font-variation-settings: 'FILL' 1"`
- **Opacity variants for borders/dividers:** always `/20` or `/50` suffix (e.g. `border-outline-variant/20`)
- **Hover transitions:** `transition-colors` or `transition-all` with `duration-150`/`duration-300`
- **Group hover patterns:** cards use `group` + `group-hover:text-primary` / `group-hover:bg-primary` for interactive lifts

### Icon set (Material Symbols Outlined)

Key icons used: `terminal`, `schema`, `person`, `architecture`, `work`, `smart_display`, `arrow_forward`, `menu`. New icons should stay within Material Symbols Outlined.

## Skills management

Use the **universal-skills-manager** skill (`/universal-skills-manager`) to discover, install, and sync skills from SkillsMP.com, SkillHub, and ClawHub into `.opencode/skills/`.

**Project-level skills directory:** `.opencode/skills/` (local to this repo, gitignored)  
**User-level skills directory:** `~/.config/opencode/skills/` (global across projects)

### Commands

- **Search:** Ask to find skills by keyword or natural language query
- **Install:** Install a skill into `.opencode/skills/` (project scope) or `~/.config/opencode/skills/` (user scope)
- **Sync:** Copy skills between different AI tool directories (Claude, Gemini, OpenCode, etc.)
- **Report:** Show a matrix of which skills are installed in which tools

### Workflow

1. Search for skills using SkillHub (no API key needed) or SkillsMP (requires `$SKILLSMP_API_KEY`)
2. Install into `.opencode/skills/` for project-specific skills, `~/.config/opencode/skills/` for global skills
3. After installation, the skill is available immediately for use in OpenCode sessions

### Workflow Sprint

1. **Planning** — @orchestrator definisce obiettivo → @pensante-architecture scrive specifiche
2. **Implementation** — @executer-implementation + @executer-test eseguono TDD
3. **Security** — @pensante-security audit → @executer-security-fix applica fix
4. **Design** — @pensante-design valida UI → @executer-ui implementa
5. **Integration** — @pensante-integration valida flussi → @executer-build + @executer-worker
6. **Commit** — @orchestrator approva → deliverable completato

## Workflow rules

0. **Safety first, little often, double check.**
1. **sequential-thinking** — Obbligatorio per ogni decisione non triviale
2. **Read PRD first.** Always read `prd.md` before implementing. Never implement features not explicitly requested. If scope is ambiguous, ask — don't assume.
4. **TDD cycle:** RED → GREEN → REFACTOR for every task. Write the failing test first, then the minimal code to pass, then refactor.
5. **Atomic commits:** one commit per functional change, Conventional Commits format with scope: `feat(news):`, `fix(worker):`, `docs:`, `test:`, `refactor:`, `chore:`.
6. **Document as you go:** when a complex bug is solved, add entry to `docs/bug_ledger.md`; when a design or architecture decision is made, update `docs/architecture.md`; track progress in `docs/progress.md`.
7. **Little often:** break work into the smallest verifiable tasks. No big-bang changes.
8. **Sprint prompts** live in `prompt/` — each sprint has its own `prompt-sprint<N>-<topic>.md` with scope, technical spec, and validation criteria.

### Bug fix flow

1. Check `docs/bug_ledger.md` for similar patterns
2. Write a test that reproduces the bug (RED)
3. Implement the fix (GREEN)
4. Refactor if needed
5. Add entry to `docs/bug_ledger.md` with symptom, cause, fix, prevention
6. Commit with `fix(<scope>):` prefix

### Commit scopes

| Scope | Covers |
|-------|--------|
| `news` | Build script, news data flow |
| `worker` | Web Worker (`newsWorker.js`) |
| `ui` | DOM rendering, layout, design system |
| `build` | Build tooling, scripts |
| `docs` | Documentation files |
| `chore` | Repo maintenance, config |

## Conventions

- **Stack:** Vanilla JS only. No React/Vue/Svelte. Zero npm dependencies for frontend runtime.
- **Tailwind:** Loaded via CDN (`cdn.tailwindcss.com`). Config is inline in `index.html` under `<script id="tailwind-config">`.
- **XSS prevention:** DOM injection must use `document.createElement` + `textContent`, never `innerHTML`.

## Web Worker constraints

- No `window` or `document` access inside `newsWorker.js` — only Worker-compatible APIs.
- Communication with main thread via `self.postMessage()` / `worker.onmessage`.
- Must handle network failures gracefully (file not found, offline) and report errors to main thread.

## Build script constraints

- `scripts/build-news.js` uses only Node native modules (`fs`, `path`). Zero npm dependencies.
- Reads `.md` files from `src/raw/`, outputs `dist/news/news-feed.json`.
- Also copies `index.html`, `main.js`, `newsWorker.js` from `src/home/` into `dist/`.
- Must be idempotent: re-running produces consistent output.
- Env vars for custom paths: `BUILD_SRC_HOME`, `BUILD_NEWS_SRC`, `BUILD_DIST`.
